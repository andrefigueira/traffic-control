# Traffic Control: Review, Research, and Gap Analysis

## Verification note

This report was produced by a multi-agent review-plus-research workflow and then spot-checked. Independently confirmed against the actual code: the `Stop` hook releases all clearances every turn (`cmd/tc/hooks.go:194`), `Heartbeat()` is exposed but called by nothing in the agent path (only a test and the API handler), and `broker.go` silently drops events to any full subscriber buffer (`internal/tower/broker.go:49`). The single most consequential strategic claim, that Anthropic ships a shared-tree multi-session coordinator that punts on file-write conflicts, was verified directly against the Claude Code "agent teams" docs (see the correction in Verdict and Industry below). The academic and historical citations, especially the 2026 arXiv entries, were not individually re-fetched, so treat their exact numbers as needing a confirming click before quoting externally. The code findings and the platform-risk framing are high confidence.

## Verdict

The core bet is sound but narrower than the current framing claims. There is a genuinely under-served niche here: a persistent, runtime, advisory coordinator over one shared working tree with live presence, a broadcast board, and first-class Claude hook plus MCP integration. The multi-agent SE literature confirms no surveyed framework occupies it (the field splits into sequential pipelines, isolate-then-merge worktree tooling, and research-stage CRDT co-editing), and the CSCW and concurrency-control canons confirm the design instincts are correct rather than novel. The single biggest risk is platform risk that is now live in the binary, though still gated: Anthropic ships Claude Code agent teams (presence panel, shared task list, mailbox, lifecycle hooks) as an experimental feature disabled by default behind `CLAUDE_CODE_EXPERIMENTAL_AGENT_TEAMS` (as of v2.1.178). Its file locking covers only task-list claim races, not source-file writes, and its sole documented guidance for source conflicts is the verbatim advisory line "Two teammates editing the same file leads to overwrites. Break the work so each teammate owns a different set of files." So the incumbent occupies the awareness layer but explicitly punts the one thing tc can own, which is enforceable file-write coordination on a shared tree. The three highest-leverage moves are: (1) fix the two correctness-of-promise bugs that make the product lie to its own agents (the flight-plan primitive locks nothing despite the SessionStart text saying it does, and the path matcher silently locks nothing under the recursive globs the README advertises); (2) make coordination fail loud instead of fail silent, since a tool whose entire value is trustworthy mediation currently hides its own degradation; (3) reposition around enforced shared-tree clearance as the non-commodity asset and treat the awareness layer as table stakes, because the awareness layer is exactly what the incumbent now ships natively.

## Code and design review

### Concurrency and lifecycle

The mutex discipline inside the tower is correct. A single `t.mu` guards all map access, `publish()` holds `t.mu` but the Broker uses an independent lock with non-blocking sends, so a stuck SSE subscriber cannot stall or re-enter the tower lock, and the delete-during-range in `expireLocked`/`releaseLocked` is legal Go. `TestConcurrentRequests` exercises the in-tower path under the race detector across 50 goroutines. The defects are at the lifecycle layer.

- **Stop hook blanket-releases all clearances every turn (CONFIRMED, downgraded critical to medium).** `hookStop` calls `c.Handoff(ctx, hookCallsign(in), "")` (cmd/tc/hooks.go:194), and an empty `releasePath` in `releaseLocked` (internal/tower/tower.go:224-240) deletes every clearance held by that callsign because the skip guard `releasePath != "" && !PathsOverlap(...)` never fires. Claude fires Stop at the end of every turn (no matcher, install_claude.go:36-38), so a hold lives only from the PreToolUse that requested it to the turn-ending Stop. The 30m `DefaultLease` whose comment (tower.go:17-24) justifies it as protection against mid-task expiry is inert: Stop, not expiry, governs hold lifetime. The verdict downgraded this from critical because the behavior is intentional and documented (hooks.go:186-187, README:46), intra-turn protection is intact, and the default mode is advisory so no hard guarantee was promised. It remains a real design and documentation inconsistency: the lease cannot be the task-spanning safety net the comment implies.

- **Heartbeat is unwired (CONFIRMED, downgraded high to medium).** `Heartbeat` (tower.go:91-109) extends `ExpiresAt` for every clearance the holder owns, is exposed via `POST /sessions/{callsign}/heartbeat` and `client.Heartbeat`, but a repo-wide grep shows the only callers are the API handler and a unit test. No hook, ticker, or CLI ever issues a heartbeat; the Serve ticker calls only `Sweep()`. The doc-comment promise that an active agent never has a path yanked is unrealized. It is downgraded because `RequestClearance` already refreshes a re-requested identical hold (tower.go:181-191) and PreToolUse re-requests on every edit, so the only uncovered case is an agent's other held paths silently expiring at 30m while it works a different file.

- **Advisory TOCTOU (CONFIRMED, downgraded high to low).** In advisory mode the tower grants both agents a clearance on the same path (tower.go:207-212) and PreToolUse only injects context (hooks.go:177-183), so the sequence A-clear, B-clear, A-write, B-write is permitted and B clobbers A across a window the tower does not control. Downgraded to low because this is the project's explicitly self-acknowledged advisory behavior with a built-in remedy (TC_ENFORCE=1 hard deny), and the recommendation reduces to documentation.

- **No queue or fairness for a blocked exclusive requester (CONFIRMED, medium).** A hard conflict returns `Granted:false` immediately (tower.go:167-176) with a deny (hooks.go:173-176). There is no wait list (grep for queue/waitlist/fifo/pending/backoff/retry returns nothing). This avoids livelock but offers no progress guarantee: since Stop hands off every turn and the holder re-requests next turn, a competing exclusive requester can be starved indefinitely. Only bites under opt-in TC_ENFORCE=1.

- **Session idle (2h) far exceeds lease (30m) (LOW).** A session quiet for 45 minutes keeps its presence row but has already lost its clearances to expiry (tower.go:328-341). Mostly academic today because Stop releases holds every turn, but it becomes the real protection ceiling if the Stop behavior is fixed.

### Protocol and path overlap

The conflict model rests entirely on `PathsOverlap`/`NormalizePath` (internal/protocol/types.go). The matcher leaks at the file level it claims to own, and these are false negatives (missed conflicts), the dangerous direction for an advisory tool.

- **Recursive globs match no nested files (CONFIRMED, high).** Go's `path.Match` has no `**` and a single `*` never crosses `/`, verified empirically: `Match("src/**","src/a/b.go")=false`, `Match("internal/**","internal/api/server.go")=false` (types.go:132-141). `dirAncestor` cannot rescue a wildcard claim because it does a literal `HasPrefix(p, "src/**"+"/")` with the `*` still in the string. The project's own test enshrines the gap (types_test.go:16), and the README advertises glob matching as a feature (README:112) while the limitations section never warns globs are single-segment only. An agent that claims `internal/**` has locked nothing below the first level; a second agent gets a clean clearance, and under TC_ENFORCE=1 the supposed hard block silently no-ops.

- **Absolute vs relative paths for the same file do not collide (CONFIRMED, downgraded high to medium).** `NormalizePath` does no repo-root anchoring, so `/repo/src/a.go` and `src/a.go` compare unequal and `PathsOverlap` returns false for one physical file. The client `relativize` (hooks.go:199-207) returns early when `cwd==""`, only relativizes when the file is under cwd, and is bypassed entirely on the MCP path (mcp.go:232 passes `a.Path` raw). Two agents launched from different cwds, or a hook caller and an MCP caller, anchor differently and miss each other. Downgraded to medium because in the default case both sessions usually launch from the same repo root, so the hole stays dormant until subdir launch, absent Cwd, or MCP usage.

- **Case-insensitive filesystems treated case-sensitively (CONFIRMED, medium).** All comparisons are case-sensitive (types.go:126, 156), so `src/App.go` and `src/app.go` produce two independent clearances on APFS (the maintainer's own platform) and default NTFS, where they are the same file. There is no `EqualFold`/`ToLower` anywhere. The blast radius is exactly the maintainer's dev environment, so it will not show up in manual testing.

- **Symlink and leading-`..` aliasing (CONFIRMED, medium).** `NormalizePath` is purely lexical (no `EvalSymlinks`/`Readlink` anywhere), so `src/a.go` and `linked/a.go` (where `linked -> src`) never overlap, and a target expressed both in-root and via an escaping `../` form does not match.

- **Unicode NFC/NFD and unconditional backslash rewrite (LOW each).** No Unicode normalization (accented filenames in different forms miss); `strings.ReplaceAll(p, "\\", "/")` fires on every platform (types.go:104), corrupting legitimate Unix filenames containing backslashes.

The one false-positive lead (the classic `src` vs `src-old` prefix bug) is **refuted**: `dirAncestor` correctly appends `/` before `HasPrefix` (types.go:156).

### Claude integration and enforcement

Every hook failure path exits 0 and allows, which is the right safety posture and is implemented consistently. The enforcement story underneath is weaker than the vocabulary suggests.

- **Advisory mode emits no machine-readable signal (CONFIRMED, downgraded high to medium).** The tower grants advisory overlaps unconditionally and the hook injects context with no `permissionDecision`. `EventConflictAlert` fires only on hard conflicts (tower.go:168); the advisory overlap branch publishes nothing, and no `advisory.overlap` event exists. So a watching human gets no live signal that two agents are on one file. Downgraded because the code is honest about being advisory by design (mcp.go:139, tower.go:125); this is an observability gap, not misrepresentation.

- **Hard mode thrashes on shared files (CONFIRMED, downgraded high to medium).** Under TC_ENFORCE=1 every request is exclusive (hooks.go:166-168) and exclusive blocks any overlap (tower.go:155), while Stop releases all holds every turn. Two agents needing one file over many turns ping-pong ownership at turn boundaries with no fairness, no queue, no backoff, and no hook-driven heartbeat to hold ground. Directory-ancestor overlap means a directory claim blocks all children. Downgraded to medium because it is confined to opt-in TC_ENFORCE=1.

- **Silent fail-open under load or partial failure (CONFIRMED, medium).** Every error path allows with no stderr warning, no board post, no degraded-mode marker (hooks.go:159-161, 169-172, 191-193). PreToolUse only Pings and never auto-starts the tower (only SessionStart does, hooks.go:80), so a crashed detached daemon stays down for the rest of the session and every edit silently proceeds uncoordinated. The Stop-on-ping-fail path also leaks holds until the 30m lease or 2h idle sweep. The system fails open and fails silent, the worst pairing for a tool whose value is trustworthy mediation.

- **Bash and MCP bypass enforcement (CONFIRMED, medium).** PreToolUse coordinates only `Edit|Write|MultiEdit|NotebookEdit` (hooks.go:141-145, install_claude.go:32). `sed -i`, heredoc redirects, `tee`, `git checkout/apply`, `gofmt -w`, `prettier --write`, `mv`/`cp`, and codemods never reach the tower, so an exclusive hold does not stop a Bash rewrite of that exact file. Separately, the MCP `request_clearance` tool reads `a.Mode` freely (mcp.go:226-232) with no server-side TC_ENFORCE floor, so a model can decline to place its own exclusive hold even under operator policy. (Per tower.go:155, it cannot punch through someone else's existing exclusive hold by downgrading, which slightly narrows the original claim.)

- **Startup race on ensureTowerRunning (MEDIUM).** Two SessionStart hooks firing together can both spawn `tc serve`; each `writePidFile()` runs before `ListenAndServe` (serve.go:27), so the loser can clobber the winner's PID, leaving the pidfile pointing at a dead process for a later `tc stop`. The 20-iteration ping loop papers over correctness for the agents, so this is a pidfile-integrity bug rather than a two-towers bug.

- **Callsign identity split (LOW).** The hook derives `claude-<12 chars of session_id>` (hooks.go:55-59) while the MCP server derives `TC_CALLSIGN` or `user@host` (mcp.go:51-56), so the same Claude session can appear under two identities and a hook-acquired hold and an MCP-acquired hold can self-conflict. The 12-char truncation itself is acceptable; the identity unification is the real issue.

### Product design coherence

- **Flight plan and clearance are decoupled (CONFIRMED, high).** `file_flight_plan` (mcp.go:207-220) only calls `PostBoard`, which appends a `BoardEntry` whose `Paths` are never passed to `PathsOverlap`. The SessionStart text fed to every agent says "post a flight plan and the tower will warn others off your files" (hooks.go:134), which is false as built: a flight plan registers no hold, runs no overlap check, and fires no `conflict.alert` when another agent later edits a declared path. This is the one finding that held at high severity through scrutiny, because the injected text is the operative instruction the product feeds every agent. (The README prose is more careful, framing flightplan as "announce what you are about to do"; the false promise is specifically in the injection.)

- **Headline clearance targets the cheap-to-recover conflict class (CONFIRMED as tension, downgraded high to medium).** `PathsOverlap` is documented as unable to model semantic coupling (types.go:117-119), and the README and ROADMAP both name the unresolved crux (README:103, ROADMAP:67). The skeptic verdict downgraded the "theatre over the easy case" framing because the README does not actually lead with the lock (the thesis statement at README:122 names presence and the board, not clearance, and README:13 calls clearance "optional"), enforcement is advisory by default, and the "same-file collision is cheap to catch via git merge" argument imports a worktree recovery model into a product whose scope is one shared tree with no inter-agent merge gate. In a shared tree, concurrent same-file writes clobber in real time with nothing to catch them, which is precisely the gap clearance fills. The tension is real and unresolved by evidence, but it is a documented trade-off the docs already weight toward the board.

- **Exclusive mode cannot protect a multi-turn task (CONFIRMED, downgraded high to medium).** Same mechanism as the Stop-release finding, viewed from the product angle: there is no mode that holds a file for the duration of a task. Downgraded because the actual write window within a turn is protected and the posture is advisory-first.

- **File granularity is a documented open question, not a defect (downgraded medium to low).** ROADMAP:66 asks file/symbol/directory/intent. The verdict noted directory and glob granularity are already a live code path (an agent can hold `auth/` via `dirAncestor` today), so "I own the auth module" is already representable. The concrete gap is narrower: board-declared flight-plan intent paths are not wired into overlap detection.

**Genuine strengths.** Single-mutex correctness with race-tested concurrency. No publish-while-locked deadlock. Deterministic conflict-holder selection by earliest `GrantedAt` (tower.go:155-166, `TestConflictHolderDeterministic`), which keeps warnings stable against Go's randomized map order. The `hasGlobMeta` gate correctly avoids the Next.js `app/[id].tsx` false-glob trap. `dirAncestor` correctly handles the `src` vs `src-old` prefix case. Hooks fail open consistently with short 2s/4s contexts so a hung tower never wedges an agent. Idempotent, non-destructive config merge into `.claude/settings.json` and `.mcp.json`. A single client type shared by CLI, hooks, and MCP so the three surfaces cannot drift. Clean SSE subscriber lifecycle with deferred unsubscribe and context-cancel, so no per-connection goroutine leak. The team is unusually honest in code comments and docs about the advisory-vs-allow trade-off, the Bash gap, and the semantic-coupling limit, which makes the product correctable.

## The research landscape

### Multi-agent LLM software engineering

The canonical frameworks almost never run multiple agents concurrently editing the same files in one shared tree. They fall into three buckets. Role-pipelined and sequential by construction: [MetaGPT (Hong et al. 2023)](https://arxiv.org/pdf/2308.00352) coordinates through a shared structured-message pool, [ChatDev (Qian et al. 2023)](https://arxiv.org/pdf/2307.07924) runs a chat-chain with one agent acting at a time, and [AutoGen (Wu et al. 2023)](https://arxiv.org/pdf/2308.08155) group chat is explicitly sequential with a manager picking one speaker. In all of these the shared state is conversational, not a mutable filesystem, so file-level concurrency control was never needed. Orchestrated parallel-but-isolated: the [OpenHands SDK (All-Hands AI 2025)](https://arxiv.org/html/2511.03690v1) uses blocking parallel delegation where sub-agents inherit workspace context but explicitly leaves locking, ownership, presence, and conflict avoidance to the developer, which is the exact layer tc provides. The strongest recent result, [Dey et al. (CAID 2026)](https://arxiv.org/html/2603.21489v1), proposes centralized asynchronous isolated delegation where each engineer agent works in its own git worktree and integrates via explicit merge, and measures that a shared workspace without isolation HURTS (57.2% to 55.5% on PaperBench). [MAGIS (Tao et al. 2024)](https://proceedings.neurips.cc/paper_files/paper/2024/hash/5d1f02132ef51602adf07000ca5b6138-Abstract-Conference.html) avoids conflicts by partitioning files in an upfront plan rather than coordinating at runtime. The one true outlier is [CodeCRDT (Pugachev 2025)](https://arxiv.org/pdf/2510.18893), which lets agents concurrently edit shared files lock-free via CRDTs, but it is simulated, toy-scale, leaves a residual semantic-conflict rate, and explicitly omits presence, leasing, and human-in-the-loop. Net: the field's dominant answer to "many agents, one repo" is isolate then merge, and a persistent advisory coordinator over one shared tree with live presence is essentially unoccupied. [Shadows in the Code (2025)](https://arxiv.org/html/2511.18467v1) adds that decentralized multi-agent pipelines are highly injectable (up to 93% attack success), so an edit-mediating broker is a natural and currently unexploited audit choke point.

### Concurrency control and leases

tc has independently rediscovered a 40-year lineage. The [lease (Gray and Cheriton 1989)](https://blog.acolyer.org/2014/10/31/leases-an-efficient-fault-tolerant-mechanism-for-distributed-file-cache-consistency/) is a time-bounded grant where the server waits out a silent holder rather than trusting it to release, and the analytic result is that the lease term should track expected access and think time, which is the formal version of tc's open heartbeat question. [Chubby (Burrows 2006)](http://muratbuffalo.blogspot.com/2010/10/chubby-lock-service-for-loosely-coupled.html) is the production realization: coarse-grained advisory locks, KeepAlive-renewed leases, plus two mechanisms tc lacks, a lock-delay quiet period before a reclaimed lock can be re-granted and sequencers (numbers handed to holders so a stale holder's late action is rejected). [ZooKeeper (Hunt et al. 2010)](https://www.usenix.org/legacy/event/atc10/tech/full_papers/Hunt.pdf) ties the lock to a session via ephemeral znodes that vanish when the heartbeat lapses, structurally identical to tc releasing on session loss, except tc binds release to turn-end rather than liveness. tc's advisory mode is recognizably [optimistic concurrency (Kung and Robinson 1981)](https://www.eecs.harvard.edu/~htk/publication/1981-tods-kung-robinson.pdf) minus the validate-and-abort step, which is exactly why it cannot prevent overwrites. [Kleppmann (2016)](https://martin.kleppmann.com/2016/02/08/how-to-do-distributed-locking.html) is the closest match to tc's stated fear: a client that pauses (GC stop-the-world at minutes, a real GitHub network delay of ~90s) has its lease expire mid-task and another client acquires it, and the answer is the fencing token plus the efficiency-vs-correctness distinction that frames tc's advisory default as an efficiency lock. [flock (Linux man-pages 2024)](https://man7.org/linux/man-pages/man2/flock.2.html) is the OS analogue: advisory, bound to the open file description, auto-released on close or death, with no timeout and no deadlock detection.

### Operational transformation and CRDTs

This body of work is the strongest available evidence FOR tc's thesis that file-level coordination, not automatic text merge, is the right layer. Both [OT (Ellis and Gibbs 1989, via the operational-transformation summary)](https://en.wikipedia.org/wiki/Operational_transformation) and CRDTs (Shapiro et al.) guarantee convergence, but neither guarantees intent. [Weidner and Kleppmann's Fugue (2023)](https://arxiv.org/pdf/2305.00583) proves both RGA and YATA (the algorithm in Yjs) can interleave concurrent insertions into incoherent output, which for code means two function bodies merging into non-parsing garbage even though the CRDT converged correctly. [Sun et al. (2019)](https://arxiv.org/pdf/1905.01518) argue convergence is the easy part and document CRDT tombstone overhead; an [industry comparison (Taskade 2026)](https://www.taskade.com/blog/ot-vs-crdt) quantifies CRDT metadata at roughly 17 to 33 times the storage. [Peritext (Litt et al. 2022)](https://dspace.mit.edu/bitstream/handle/1721.1/147641/3555644.pdf) shows plain sequence CRDTs lose intent on structured content, and [Eg-walker (Gentle and Kleppmann 2025)](https://arxiv.org/abs/2409.14252) is the modern hybrid to evaluate if text-level merge were ever pursued. The load-bearing source is again [CodeCRDT (Pugachev 2025)](https://arxiv.org/pdf/2510.18893): applying a Yjs CRDT to multiple Claude agents co-editing code gives 100% convergence and zero merge failures, yet roughly 5 to 10% of concurrent edits are semantic conflicts (duplicate declarations, type mismatches, broken references) that converge cleanly and are wrong, rising to 80% on complex coupled tasks, with parallel agents producing measurably lower code quality (about -7.7%). This is direct empirical evidence that auto-merge does not solve the agent co-editing problem at the level that matters, and it answers the naive objection ("why lock a file when Google Docs lets people co-edit a line?").

### Semantic and continuous merge-conflict detection

The cross-path coupling tc disclaims is the well-studied higher-order conflict problem. Structured merge ([jDime 2012](https://www.se.cs.uni-saarland.de/publications/docs/LAL15ase.pdf), [IntelliMerge 2019](https://dl.acm.org/doi/10.1145/3360596), [Spork 2022](https://arxiv.org/abs/2202.05329), [Mergiraf 2025](https://lwn.net/Articles/1042355/), [weave](https://github.com/ataraxy-labs/weave), [gfix](https://www.gfix.space/)) is within-file and post-hoc, the wrong layer for proactive cross-file warning. The transferable lineage is proactive and continuous: [Crystal (Brun et al. 2011)](https://cs.uwaterloo.ca/~rtholmes/papers/fse_2011_brun.pdf) is a background daemon that speculatively merges, builds, and tests peers' states, classifying TEXTUAL, BUILD, and TEST conflicts, and BUILD and TEST are exactly the signature-vs-caller breakage tc cannot see, caught cheaply by reusing the compiler and test suite. [WeCode (Guimaraes and Silva 2012)](https://www.researchgate.net/publication/254040877_Improving_early_detection_of_software_merge_conflicts) pushes this into the IDE. [Owhadi-Kareshk et al. (2019)](https://arxiv.org/abs/1907.06274) show a cheap ML filter predicts conflict-free merges with high recall (~0.96) and modest conflict-class precision (~0.63), usable as a gate before expensive speculative build. Static interference detection ([De Jesus, Borba et al. 2023](https://arxiv.org/html/2310.04269)) directly targets different-files-breaking-each-other via interprocedural data-flow and PDG analysis, but at ~17.8s per unit, Java/Soot-bound, and false-positive-prone (precision ~0.43). The 2026 agent-specific work confirms the problem is acute: [CooperBench (Khatua et al. 2026)](https://arxiv.org/pdf/2601.13295) finds a "curse of coordination" where agents score ~30% lower when cooperating on potentially-conflicting features, rooted in vague messages, broken commitments, and wrong models of peers, which validates tc's board bet while warning that a passive board is insufficient. [AgenticFlict (Ogenrwot and Businge 2026)](https://arxiv.org/pdf/2604.03551) measures a 27.67% textual conflict rate across 142,652 agentic PRs and explicitly flags semantic, build, and test conflicts as an open dataset gap.

### CSCW workspace awareness

tc's board and scope are a direct instantiation of awareness theory. [Dourish and Bellotti (1992)](https://www.dourish.com/publications/1992/cscw92-awareness.pdf) found that awareness picked up passively through the shared workspace beats dedicated explicit information-generation mechanisms that impose a production cost and fragment attention. [Gutwin and Greenberg (2002)](https://collablab.northwestern.edu//CollabolabDistro/nucmc/GutwinGreenberg_FrameworkWorkspaceAwareness.pdf) give the audit vocabulary (Who/What/Where/When, maintained via consequential communication, feedthrough, and intentional communication) and the radar-view and per-actor-colour widgets. [Heath and Luff (1992)](https://www.researchgate.net/publication/242317961_Collaboration_and_Control_Crisis_Management_and_Multimedia_Technology_in_London_Underground_Line_Control_Rooms) show effective awareness is peripheral and designed to be noticed, and [Schmidt (2002)](https://www.researchgate.net/publication/2546797_The_Problem_with_'Awareness'_Introductory_Remarks_on_'Awareness_in_CSCW') warns awareness is a skilled accomplishment, not data to dump. A [hybrid-work design-space paper (DIS 2024)](https://dl.acm.org/doi/fullHtml/10.1145/3643834.3661549) restates the privacy-vs-awareness and awareness-vs-disturbance trade-offs and the persistence-reduction mitigation. Mapped onto tc: the scope dashboard is a textbook radar view and is theory-aligned, while the SessionStart injection is the opposite design point, an explicit one-shot push delivered before the agent knows what it will touch, going stale immediately and spending context-window budget on global state. The PreToolUse advisory injection is the better-designed channel because it is feedthrough at the point of action. [CooperBench (2026)](https://arxiv.org/pdf/2601.13295) independently rediscovers the CSCW problem for agents, and a [multi-agent collaboration survey (2025)](https://arxiv.org/pdf/2501.06322) frames where tc sits among coordination structures. An open intersection: the entire awareness canon assumes a human perceiver with cheap peripheral vision, while an LLM pays tokens per line of injected awareness and cannot glance mid-turn, which reframes overload as token budget and argues for pull over front-loaded push.

### Industry parallel-agent state of the art

The 2024-2026 industry has converged hard on isolation. [Cursor 2.0 (2025)](https://www.techzine.eu/news/devops/135916/cursor-2-0-introduces-parallel-agents-and-new-model/) runs up to 8 agents each in a git worktree, [GitHub Copilot coding agent (2025)](https://github.blog/news-insights/product-news/github-copilot-meet-the-new-coding-agent/) runs each task in a cloud Actions environment and returns a PR, and [Devin 2.0 (Cognition 2025)](https://cognition.ai/blog/devin-annual-performance-review-2025) runs parallel instances each in an isolated VM (PR merge rate 34% to 67% year over year). A whole [sandbox-per-agent infrastructure layer (2026)](https://agentmarketcap.ai/blog/2026/04/07/ai-agent-sandbox-infrastructure-e2b-modal-daytona-fly-machines-secure-code-execution) exists (E2B, Daytona, Modal, Fly). Coordination protocols are consolidating under the Linux Foundation with [A2A (2025)](https://www.linuxfoundation.org/press/linux-foundation-launches-the-agent2agent-protocol-project-to-enable-secure-intelligent-communication-between-ai-agents) and [AGNTCY (2025)](https://www.linuxfoundation.org/press/linux-foundation-welcomes-the-agntcy-project-to-standardize-open-multi-agent-system-infrastructure-and-break-down-ai-agent-silos). The most consequential finding is that [Anthropic's Claude Code agent teams](https://code.claude.com/docs/en/agent-teams) ships a shared-tree multi-session coordinator (shared task list, mailbox, presence panel, TeammateIdle/TaskCreated/TaskCompleted hooks), and [Anthropic now ships native worktree isolation](https://code.claude.com/docs/en/agents) including declarative `isolation:worktree` for subagents. Two clarifications confirmed against the live docs: agent teams is experimental and disabled by default behind `CLAUDE_CODE_EXPERIMENTAL_AGENT_TEAMS` (v2.1.178), and its "file locking" guards only task-claim races on the shared task list, not source-file writes. Critically, agent teams does NOT isolate teammates in worktrees and its documented file-conflict mitigation is purely advisory ("Two teammates editing the same file leads to overwrites. Break the work so each teammate owns a different set of files."), and [Anthropic's own multi-agent research post (2025)](https://www.anthropic.com/engineering/multi-agent-research-system) states they avoided shared live state because "asynchronicity adds challenges in result coordination, state consistency, error propagation." The platform risk in tc's self-assessment is now active, not hypothetical, but the incumbent left open the precise piece tc can own.

### Direct prior art

tc's advisory file coordination is table stakes, not a moat. The de-facto pattern across [agent-comms (2025)](https://dev.to/lakshmisravyavedantham/i-built-a-coordination-system-for-multiple-claude-ai-agents-so-they-stop-overwriting-each-other-448i), [claude_code_agent_farm (2025)](https://github.com/Dicklesworthstone/claude_code_agent_farm), [mcp_agent_mail (2025)](https://github.com/dicklesworthstone/mcp_agent_mail), [agent-hub-mcp (2025)](https://github.com/gilbarbara/agent-hub-mcp), and Anthropic agent teams is file-based or message-based advisory coordination, all poll-based and stateless-per-call. `mcp_agent_mail` is the closest direct competitor (advisory file reservations with TTL, conflicts reported-but-granted, an optional pre-commit guard, SQLite persistence) but is HTTP-only, poll-based with no live presence, and pure MCP with no hook integration. On the human-tooling side, pessimistic locking has a 25-year cautionary track record: [SVN lock-modify-unlock (SVN Book)](https://svnbook.red-bean.com/en/1.7/svn.advanced.locking.html) is documented to fail silently ("the first you know is when your commit fails"), and [git-lfs locking](https://github.com/git-lfs/git-lfs/blob/main/docs/proposals/locking.md) is deliberately scoped to unmergeable binaries only. [CODEOWNERS](https://docs.github.com/en/repositories/managing-your-repositorys-settings-and-features/customizing-your-repository/about-code-owners) validates the advisory-default plus enforced-on-opt-in spectrum, and [VS Code Live Share](https://code.visualstudio.com/blogs/2017/11/15/live-share) confirms live presence alone does not prevent overwrites, so presence must be coupled with clearance. The one published pro-locking argument for the agent era, [Perforce (2025)](https://www.perforce.com/blog/vcs/p4-vs-git-for-ai-coding-agents), reasons that agents hit the merge wall faster than humans, but carries zero quantitative evidence. The strongest case for tc is that agents defang locking's historical failures (no ego about waiting, programmatically force-releasable, sub-minute cycles, they read injected context), and tc's defensible combination is the persistent daemon with SSE push instead of poll, enforcement at the PreToolUse boundary earlier than any pre-commit guard, and a single zero-dep Go binary that is CLI, hook, and MCP at once.

### Harden path normalization

**What it is.** Close the confirmed false negatives inside the file-level model: add `**` support (or rewrite trailing `/*` and `/**` into directory claims that route through `dirAncestor`), anchor abs vs rel canonically in the tower against the known project root so the hook and MCP entry points cannot diverge, case-fold on darwin and windows, and `EvalSymlinks` to a canonical form or document the alias as unsupported.

**Why it matters.** These are missed conflicts, the dangerous direction for an advisory tool, and they are most dangerous under TC_ENFORCE=1 where the supposed hard block silently no-ops. The recursive-glob hole is especially bad because the README advertises glob matching while a recursive subtree claim locks nothing below the first level.

**Prior art.** None external; these are internal correctness bugs in tc's own matcher. The fix parts are all commodity stdlib or a ~40-line hand-rolled matcher: `doublestar`-style matching, `filepath.Abs`/`Rel`, `strings.EqualFold`, `filepath.EvalSymlinks`. Anthropic's `isolation:worktree` sidesteps the whole overlap problem rather than fixing the matcher, so it is not prior art here.

**Feasibility.** High for the matcher fixes, all localized to `protocol/types.go` and guarded by `types_test.go`. One honest correction surfaced under scrutiny: the report's claim that the tower already knows the project root is wrong. `RequestClearance` only ever receives `NormalizePath(reqPath)` and the tower learns only the project basename at register time (`Session.Project`), so true tower-side anchoring means threading cwd or an absolute root through the clearance wire format, which is more than a two-file change. The maintainer's own claimed precedent (`scope.go:31-34` doing GOOS-conditional case handling) also does not exist; that code is `openBrowser` choosing `open`/`rundll32`/`xdg-open`.

**Adversarial verdict: pursue.** The prior-art lens returned pursue: the work is trivial and unambiguous, and only tc can fix tc's matcher, so prior art is the parts list rather than a competing product. The feasibility lens returned maybe, with the strongest objection that most of these false negatives require two agents editing the same physical file via different spellings within the same single turn (near-zero given Stop releases holds every turn), so only the abs/rel-plus-MCP-raw-path slice meaningfully moves real-collision prevention while the rest is internal-consistency polish. Resolution: pursue the recursive-glob fix and the abs/rel-plus-MCP canonicalization as the high-value core (the glob fix because it makes an advertised feature actually work, the abs/rel fix because the MCP entry point genuinely misses on the live path), and treat case-fold and symlink handling as cheap correctness hygiene worth bundling since the stdlib calls are trivial, while accepting that tower-side anchoring carries a real wire-format cost the maintainer should scope before committing.

### Fail-loud observability and reliable event delivery

**What it is.** Two coupled holes. The hook fail-silent path (every error returns 0 with no signal, and PreToolUse never auto-starts a crashed tower): add a stderr warning on degrade, call `ensureTowerRunning` from PreToolUse, and reconcile stale self-holds on next contact. The broker silent drop (`Publish` discards to any full 64-deep subscriber buffer with no counter, log, or sequence marker, and `tc watch` exits 0 quietly on stream drop while the browser dashboard auto-reconnects): add per-subscriber drop counters surfaced in health, and a client reconnect-with-backoff loop.

**Why it matters.** For a tool whose entire value is trustworthy mediation, failing open and silent means the operator cannot tell coordination is gone. The Stop-releases-each-turn dynamic strengthens this: when enforcement is weak and ephemeral, awareness is the only residual value, so a silently degraded awareness channel is the worst failure.

**Prior art.** No external tool; this is standard fail-loud engineering simply absent from the codebase. SVN's documented worst failure mode is exactly silent lock failure, which the research frames push-presence as the fix for. The browser `EventSource` already auto-reconnects, making the CLI's lack of it a real internal inconsistency.

**Feasibility.** High and cheap. Stderr writes, a PreToolUse `ensureTowerRunning` call, a per-subscriber dropped-count int, and a reconnect loop in `client.Events` are small zero-dep changes.

**Adversarial verdict: pursue.** The feasibility lens returned pursue with one important refinement: part one (PreToolUse auto-start plus stderr-on-degrade) is the strong high-value half because it is a genuine asymmetry bug that directly restores the mediation promise when the daemon crashes. Part two is overstated, because the load-bearing collision-prevention path is the synchronous `RequestClearance` RPC against authoritative in-memory state, not the lossy broadcast broker. A dropped broker event degrades a dashboard, not the actual gate. Resolution: pursue the cheap core (PreToolUse auto-start, stderr-on-degrade, a broker drop counter in `tc health`, and watch reconnect-with-backoff), and defer the full sequence-number-plus-gap-marker-plus-resync protocol as gold-plating relative to its sole consumers (a debug watch and a self-reconnecting browser).

### Fix the flight-plan / clearance disconnect

**What it is.** Wire the announce-intent primitive to the conflict engine so the SessionStart promise becomes true, or correct the injected text. `file_flight_plan` only calls `PostBoard`, whose `Paths` never reach `PathsOverlap`, so the injection "post a flight plan and the tower will warn others off your files" is false as built. Paired suggestion: reposition the product to lead with the awareness layer and demote the lock to optional.

**Why it matters.** The injected text is the operative instruction fed to every agent and it oversells a mechanism that holds nothing, a concrete correctness-of-promise bug.

**Prior art.** [CodeCRDT (2025)](https://arxiv.org/pdf/2510.18893) and [CooperBench (2026)](https://arxiv.org/pdf/2601.13295) independently validate the board-plus-claims-plus-observe pattern, and CooperBench's curse of coordination warns a passive board is too weak, favoring active targeted warnings.

**Feasibility.** High for the wiring (route `BoardEntry.Paths` through the existing overlap scan, or fix the one injected string) and zero cost for the framing.

**Adversarial verdict: pursue the bug fix; invert the reposition.** Both lenses agreed the bug is real, narrow, and prior-art-proof (you cannot have prior art for your own broken string). The strongest objection lands on two sub-claims. First, the repositioning thesis markets directly into the incumbent's feature set: the awareness layer it wants to lead with is exactly what agent teams now ships, and CooperBench says a passive board underperforms, so the board is the commoditized part and the instinct to demote the lock is backwards relative to the evidence. The defensible wedge is enforced shared-tree file-write clearance, which agent teams explicitly tells users to handle manually. Second, the wiring half stays advisory (injected context with no permission decision) and risks noise, since flight-plan paths are deliberately coarse and `dirAncestor` matching would fire warnings across a large fraction of every other agent's per-edit scans. Resolution: fix the false promise unconditionally (correct the text and align the docs), and ship the wiring only with care for path breadth, framed honestly as a durability and awareness improvement (flight-plan board entries are the only path-bearing primitive that survives Stop, since clearances are released every turn) rather than as collision prevention. Reposition around enforced clearance as the non-commodity asset and treat awareness as table stakes that even worktree users would value.

### Symbol-graph clearances

**What it is.** Generalize `PathsOverlap` from path/glob/dir-ancestor to symbol overlap: a lightweight exported-symbol-and-reference index (ctags-style or tree-sitter) keyed per file, so when agent A's clearance touches a file defining symbol S and another live clearance is on a file referencing S, the tower emits an advisory coupling warning. It can also gate a more expensive check.

**Why it matters.** Both the review (types.go:117-119 concedes semantic coupling is out of scope) and the entire research corpus name this as the durable differentiator. CodeCRDT measured a 5 to 10% (up to 80%) semantic-conflict rate; worktree isolation and agent teams both punt logical conflicts to merge or CI. It is defensible against platform risk because the incumbent does not attempt it.

**Prior art.** [De Jesus and Borba (2023)](https://arxiv.org/html/2310.04269) do interprocedural interference detection but heavyweight, Java-bound, and noisy. The index technology (ctags, tree-sitter, LSP, aider repomap) is commodity. The novelty is the composition: feeding the index into a live clearance broker over one shared tree. Augment Code does post-implementation semantic cross-dependency verification, but paired with worktree isolation and timed before PR, not as a live pre-edit advisory on a shared tree.

**Feasibility.** Medium-high for a coarse index; tree-sitter or LSP raises fidelity but adds a dependency that cuts against the single-binary ethos. Ship advisory-only, never as a hard block.

**Adversarial verdict: maybe.** The prior-art lens returned maybe: the gap is real but narrower than framed, since Augment effectively does symbol coupling from the verification side, so the defensible wedge is timing (live, pre-edit) and shared-tree placement, not "nobody does symbol coupling," and the moat is months not years. The feasibility lens returned maybe with a sharper structural objection: the temporal model can defeat the trigger. A coupling warning only fires if, at the instant A requests clearance on the file defining S, B holds a live clearance on a file referencing S, but Stop releases all clearances at turn end, so the dominant real case is staggered (B edited and released the caller ten minutes ago) and at that moment no live clearance exists to detect. It can only see concurrently-held coupled files. The objection also notes the channel is ignorable and a coarse regex index floods false positives. Resolution: this is the most strategically valuable direction in the report and worth prototyping, but only if it is anchored on the persistent board and SessionStart injection (a "B recently touched a file referencing S" hint that survives Stop) rather than on the turn-scoped clearance broker, and only after the path-hardening and observability work prove out. Build the cheap symbol index as a board-level hint first and measure false-positive rate before investing further.

### Liveness-driven leases

**What it is.** Replace the two broken halves of the lease story with one model grounded in session liveness: on Stop release only paths marked done, drive renewal from a background liveness signal independent of tool calls, and bind a clearance's lifetime to the holding session's liveness (the ZooKeeper ephemeral pattern) so a crashed agent's holds drop fast and a live-but-quiet agent's holds persist.

**Why it matters.** It addresses the most load-bearing lifecycle defect and the lease-expiry-mid-task pain that the research shows is the shared unsolved problem across every competitor (agent-comms 5min, agent_farm 2hr, mcp_agent_mail ttl).

**Prior art.** [Gray and Cheriton (1989)](https://blog.acolyer.org/2014/10/31/leases-an-efficient-fault-tolerant-mechanism-for-distributed-file-cache-consistency/), [Chubby (2006)](http://muratbuffalo.blogspot.com/2010/10/chubby-lock-service-for-loosely-coupled.html), [ZooKeeper (2010)](https://www.usenix.org/legacy/event/atc10/tech/full_papers/Hunt.pdf), and flock.

**Feasibility.** The tower stores `Session.PID` and `LastSeen` and the Serve ticker already sweeps every 30s, so the wiring exists. But the dominant hook flow registers the PID of the transient `tc hook` process, which exits in milliseconds and may be recycled, so a signal-0 probe would mark hook-born sessions dead almost instantly; only the MCP path registers a durable PID, and that is not where clearances originate.

**Adversarial verdict: drop on novelty, maybe as engineering.** The prior-art lens returned drop: the proposal dresses a textbook fix in novel-sounding language while its own claimed-prior-art field lists every system that already solves it. ZooKeeper has bound holds to session-liveness fact (not a TTL guess) for 15 years, so the headline claim of being the only tool to do so is false. The single hedge ("none reason about LLM think-time") does not survive contact, because "alive but not issuing lockable operations" is exactly the holder state ZooKeeper and Chubby invented a separate keepalive channel for. The feasibility lens returned maybe and identified the wrong liveness anchor (the hook PID) plus two further problems: the value is mostly off by default (cross-turn extension only bites under opt-in TC_ENFORCE=1), and releasing only explicitly-done paths risks more false blocking because LLMs rarely call the handoff tool, so a touched-once-then-abandoned file would block others until liveness expiry, discarding the self-cleaning virtue of the current turn-scoped model. Resolution: do not pursue this as a novel feature, and do not adopt the "release only done paths" half. If the maintainer wants to fix the documented lease inconsistency, the honest move is small engineering: either wire a durable per-agent keepalive (SessionStart or MCP-driven, not the hook PID) so an agent's other held paths do not silently expire at 30m, or simply drop the 30m-lease pretense in the docs and acknowledge turn-scoped holds as the deliberate design. The latter is cheaper and matches what the code actually does.

### Region/symbol-level sub-file clearances

**What it is.** Let an agent claim a line range or named symbol within a file so two agents editing different functions in one file are not forced to serialize.

**Why it matters.** The OT/CRDT literature shows disjoint positions in one file can be edited safely, and finer granularity reduces the false conflicts that are the dominant cost of coarse locking.

**Prior art.** Regional-exclusion patents and OT/CRDT position identifiers solve the locking technique; no shared-tree agent coordinator exposes line or symbol-range clearances. VS Code Live Share is not prior art (it does OT co-editing with no locking). ROADMAP:66 names the granularity question.

**Feasibility.** The data-model change is small, but raw line ranges drift as files change so it only works symbol-anchored, which depends on the unbuilt symbol index.

**Adversarial verdict: maybe then drop.** The prior-art lens returned maybe: genuinely unoccupied for tc's case, but the kill shot is value not novelty, because tc is advisory by default (two agents editing disjoint functions already do not corrupt each other, they both just get warned), the symbol-anchoring dependency inflates cost, and the ROADMAP's own validation experiment says if cross-path breakage dominates the board matters far more than finer clearance, so this optimizes the rarest collision class. The feasibility lens returned drop outright: Stop releases all holds every turn so whole-file holds already only live one turn, advisory is default, a sub-file claim forces a synchronous file read and old-string-to-symbol resolution on every edit, line ranges grant false safety, and it does not address the dominant cross-path breakage. Resolution: drop until the validation experiment proves same-file disjoint contention is a real and frequent problem, which the evidence suggests it is not.

### Out-of-band edit reconciliation (Bash bypass)

**What it is.** An optional tower-side filesystem watcher per project root that reconciles actual on-disk writes against held clearances and publishes an unattributed-write or conflict event, catching Bash mutators, codemods, and non-Claude tools.

**Why it matters.** The review and three research angles name this as open space no competitor addresses, and it is feasible because the tower already runs as a daemon.

**Prior art.** Anthropic's own hooks guide documents the mechanism (a PostToolUse Bash matcher running `git status --porcelain`, plus a Stop hook that scans the tree per turn) framed for audit logging. No competitor does cross-agent attribution of an out-of-band write.

**Adversarial verdict: maybe then drop.** The prior-art lens returned maybe and corrected the central "open space, moat" claim: the documented git-status idiom already covers single-session detection, and since tc is already the hook handler it could add a Bash case to the matcher rather than build a heavier fsnotify reconciler. The only thin novelty is cross-agent attribution of an unattributed write, which is a thin wrapper over `git diff` plus the existing clearance map. The feasibility lens returned drop on the consumption path, which is the part that matters: there is no PostToolUse, Notification, or any hook that pushes into a live agent's turn, and SSE is read by a human dashboard, not by the offending agent, so an alert lands somewhere the offender will not see until its next SessionStart by which point the clobbering write already happened. It detects what it cannot prevent and alerts an audience that is not listening during the turn. The Stop-releases-each-turn dynamic also collapses the window where a hold exists to reconcile against, and the common case (every gofmt, git checkout, build artifact) is "changed with no clearance," producing poor signal-to-noise. Resolution: drop the watcher. If the maintainer wants to catch Claude-issued Bash mutators at the point they can act, the architecture-aligned move is a PostToolUse Bash hook that parses or diffs after Bash runs, not a tree watcher. A post-hoc forensic audit log is a few hours of work if desired, but it is observability, not coordination.

### Speculative build/test on concurrent clearances

**What it is.** When two clearances are concurrently active over coupled code, have the tower run a background build and the affected tests against the union of both agents' in-flight edits and surface BUILD/TEST conflicts on the board, reusing the project's own compiler rather than building semantic analysis.

**Why it matters.** It is the best-precedented technique for the cross-path coupling tc disclaims, and the tower is already architecturally Crystal's daemon minus the speculative loop.

**Prior art.** [Crystal (Brun et al. 2011)](https://cs.uwaterloo.ca/~rtholmes/papers/fse_2011_brun.pdf) and [WeCode (2012)](https://www.researchgate.net/publication/254040877_Improving_early_detection_of_software_merge_conflicts) do speculative merge-build-test for human branches; the uncommitted-changes variant is patented and shipped commercially (GitLive); [Owhadi-Kareshk (2019)](https://arxiv.org/abs/1907.06274) and [jDime (2012)](https://www.se.cs.uni-saarland.de/publications/docs/LAL15ase.pdf) provide the cheap-then-expensive gating idea.

**Adversarial verdict: drop.** Both lenses returned drop. The prior-art lens found the distinguishing move (speculatively building the union of two agents' uncommitted edits and warning before commit) is patented and a live commercial feature for human teammates, so swapping in agents is a re-skin, not a new mechanism, with non-trivial patent exposure. The feasibility lens found four architecture-fatal facts: the tower stores only path strings and never sees file content, so there are no in-flight edits to union; because all agents share one tree there is no version separation to union (importing it reintroduces the worktree isolation tc bet against); Stop releases holds every turn so the concurrently-held window is sub-second while a build takes seconds to minutes, meaning the signal arrives after the window it was meant to gate; and building in a live shared tree fails on any agent's half-written file. The gap's own text concedes it should ship only after symbol-graph clearances which "deliver most of the value far cheaper," an argument against itself. Resolution: drop. Wrong layer, wrong architecture, late signal, and patent exposure.

## Recommended sequence

1. Fix the false promise in the SessionStart flight-plan text (correct the string and align the docs). It is the cheapest change in the report and stops the product lying to every agent it touches.
2. Fix the recursive-glob hole and the abs/rel-plus-MCP path canonicalization, bundling case-fold and symlink hygiene since the stdlib calls are trivial. This makes an advertised feature actually work and closes the live-path miss between hook and MCP callers. Scope the tower-side anchoring wire-format change before committing to it.
3. Make coordination fail loud: add a PreToolUse `ensureTowerRunning` call and a stderr warning on degrade, surface a broker drop counter in `tc health`, and add reconnect-with-backoff to `tc watch`. Defer the full event-sequence-plus-resync protocol.
4. Resolve the lease inconsistency the cheap way: either wire a durable SessionStart or MCP keepalive (not the hook PID) or drop the 30m-lease pretense in the docs and own turn-scoped holds as the design.
5. Reposition: lead with enforced shared-tree clearance as the non-commodity asset that agent teams punts on, treat the awareness layer as table stakes, and run the ROADMAP:67 validation experiment (instrument a real two-agent session, count same-path vs cross-path breakage) before any further investment in finer locking.
6. Prototype the symbol-graph coupling hint as a board-level signal that survives Stop, advisory-only, and measure its false-positive rate before deciding whether it earns a place. This is the most strategically defensible direction against platform risk, but only if it lands on the persistent board rather than the turn-scoped clearance broker.

Do not build: liveness-driven leases as a novel feature (textbook prior art, wrong PID anchor), sub-file clearances (optimizes the rarest collision class), the filesystem-watch reconciler (detects what it cannot prevent, no consumption path), or speculative build/test (wrong architecture, patent exposure).
